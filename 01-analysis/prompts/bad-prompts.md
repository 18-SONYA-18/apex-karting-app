
# «Плохие» промпты (антипримеры)

> Для лекции. 2–3 промпта, где ИИ выдаёт мусор: шаблонные user stories не из домена,
> выдуманные интеграции/стандарты, путаница ФТ/НФТ. Показываем для контраста.

---

## Антипример 1: «Напиши ТЗ на картинг»

**Промпт:**
> Напиши мне ТЗ на мобильное приложение для картинг-центра. Чтобы клиенты могли записываться на заезды, выбирать экипировку и платить онлайн.

**Что не так:**
- Нет контекста: какой картинг-центр, какие ограничения, что уже есть
- Нет ролей: кто пользователь, кто админ
- Нет скоупа: что входит в MVP, что потом
- ИИ начинает выдумывать: онлайн-оплата (которой нет в брифе), рейтинги, программа лояльности

**Ответ ИИ (фрагмент):**
> **User Stories:**
> - Как клиент, я хочу оплачивать заезд через Apple Pay, чтобы не носить наличные
> - Как маршал, я хочу видеть расписание в реальном времени
> - Как владелец, я хочу получать push-уведомления о бронях
>
> **Интеграции:**
> - Stripe для онлайн-оплаты
> - Firebase для push-уведомлений
> - Google Maps API для построения маршрута до центра

**Как доводить:**
- Дать бриф заказчика с ограничениями (офлайн-оплата, только роль клиента, MVP)
- Указать, что уже есть (существующая админка, инфраструктура)
- Попросить не выдумывать интеграции, а работать с тем, что есть

## Антипример 2: «Сделай user stories для бронирования»

**Промпт:**
> Напиши user stories для функции бронирования заезда в картинг-центре.

**Что не так:**
- User stories получаются шаблонными, не из домена картинга
- Нет критериев приёмки
- Нет связи с функциональными требованиями
- ИИ путает ФТ и НФТ (например, «быстрая загрузка» — это НФТ, а не ФТ)

**Ответ ИИ (фрагмент):**
> **User Stories:**
> - Как пользователь, я хочу быстро загружать приложение, чтобы не ждать
> - Как клиент, я хочу видеть список заездов
> - Как админ, я хочу управлять расписанием
>
> **Нефункциональные требования:**
> - Приложение должно поддерживать 1000 одновременных пользователей
> - Время отклика API < 100 мс

**Как доводить:**
- Дать контекст домена (картинг, экипировка, маршалы, правило 2 часов)
- Попросить критерии приёмки в формате Given/When/Then
- Разделить ФТ (что делает система) и НФТ (как работает)
- Указать, какие роли есть в системе (только клиент, без админа в MVP)

## Антипример 3: «Опиши API для бронирования»

**Промпт:**
> Опиши REST API для мобильного приложения картинг-центра. Нужны эндпоинты для бронирования, отмены и оплаты.

**Что не так:**
- ИИ выдумывает эндпоинты, которых нет в реальности
- Добавляет оплату (которой нет в скоупе)
- Не учитывает существующую инфраструктуру
- Путает методы HTTP (например, DELETE для отмены вместо POST)

**Ответ ИИ (фрагмент):**
> **Endpoints:**
> - `POST /api/v1/bookings` — создать бронь
> - `DELETE /api/v1/bookings/{id}` — отменить бронь
> - `POST /api/v1/payments` — оплатить бронь через Stripe
> - `GET /api/v1/tracks` — получить список трасс
> - `POST /api/v1/reviews` — оставить отзыв после заезда
>
> **Authentication:**
> - JWT токены с refresh через `/auth/refresh`
> - OAuth 2.0 для входа через Google/Apple

**Как доводить:**
- Дать существующую спецификацию API (OpenAPI)
- Указать, что уже реализовано (auth, profile, slots, bookings)
- Попросить не выдумывать эндпоинты, а работать с тем, что есть
- Уточнить, какие методы используются (POST для отмены, не DELETE)

---

## Общие правила хороших промптов

1. **Контекст.** Дай бриф, ограничения, что уже есть.
2. **Роли.** Кто пользователь, кто админ, кто в скоупе.
3. **Скоуп.** Что входит в MVP, что потом.
4. **Не выдумывать.** Работай с тем, что есть, не добавляй интеграции/функции из головы.
5. **Критерии.** Проси критерии приёмки, а не просто описания.
6. **Разделяй.** ФТ (что делает) и НФТ (как работает) — разные вещи.
