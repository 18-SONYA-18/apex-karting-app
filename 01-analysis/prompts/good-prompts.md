
# «Хорошие» промпты

> Для лекции. Примеры, где контекст важнее промпта: даём домен, ограничения, пользователей —
> и просим уточняющие вопросы / структурированный артефакт.

---

## Пример 1: Функциональные требования с контекстом домена

**Промпт:**
> Я системный аналитик, работаю над мобильным приложением для уличного картинг-центра «Апекс».
>
> **Контекст:**
> - Клиенты записываются на заезды через приложение (заменяет запись через Telegram и доску маркером)
> - Оплата только офлайн (наличные/перевод), онлайн-оплаты нет
> - Только роль «Клиент» в приложении, маршалы и владелец работают через существующую админку
> - Экипировка (шлем+подшлемник) бывает своя или прокатная
> - Правило 2 часов: отмена ≥2ч до старта освобождает место, <2ч — нет, но штрафов нет
>
> **Задача:** Напиши функциональные требования для сценария «Запись на заезд» (UC-1).
>
> **Формат:**
> - FR-XXX: [название] — [описание]
> - Для каждого FR укажи приоритет (Must/Should/Could)
> - Не выдумывай интеграции, работай только с тем, что есть
> - Если чего-то не хватает в контексте — задай уточняющие вопросы
>
> **Ограничения:**
> - Не добавляй онлайн-оплату, рейтинги, программу лояльности
> - Лимит мест на бронь: 4 (себя + до 3 гостей)
> - Два независимых лимита: места и прокатная экипировка

**Почему работает:**
- Даёт полный контекст домена (картинг, экипировка, правило 2 часов)
- Указывает ограничения (офлайн-оплата, только клиент, без выдумок)
- Задаёт формат вывода (FR-XXX с приоритетами)
- Просит уточняющие вопросы, если чего-то не хватает
- Явно запрещает выдуманные функции

**Ответ ИИ (фрагмент):**
> **Уточняющие вопросы:**
> 1. Какое максимальное количество прокатных комплектов экипировки в центре? (нужно для FR-14)
> 2. Есть ли разные конфигурации трасс (новичковая/опытная) с разными потолками мест?
> 3. Нужно ли показывать клиенту причину отмены заезда маршалом?
>
> **Функциональные требования (UC-1: Запись на заезд):**
>
> **FR-10** (Must): Система должна позволять клиенту выбрать слот для записи из списка доступных заездов.
>
> **FR-11** (Must): Система должна позволять клиенту выбрать вариант экипировки (своя/прокатная) для каждого места в брони.
>
> **FR-12** (Must): Система должна ограничивать число мест в одной брони значением 4 (себя + до 3 гостей).
>
> **FR-13** (Must): Система должна проверять доступность мест по формуле: `max_seats = min(free_seats, track_config.capacity_cap, 4)`.
>
> **FR-14** (Must): Система должна учитывать прокатную экипировку как отдельный лимит: `rental_count ≤ free_rental_equipment`.
>
> **FR-15** (Must): Система должна запрещать запись сверх лимита мест или прокатной экипировки и предотвращать овербукинг.

## Пример 2: User stories с критериями приёмки

**Промпт:**
> Контекст: мобильное приложение «Апекс» для картинг-центра. Клиент записывается на заезды.
>
> **Домен:**
> - Слот = заезд (дата, время, конфигурация трассы, маршал, цена, места)
> - Экипировка: своя или прокатная (шлем+подшлемник)
> - Отмена: ранняя (≥2ч, место освобождается) / поздняя (<2ч, место не освобождается, штрафов нет)
>
> **Задача:** Напиши user stories для сценария «Отмена записи» (UC-2).
>
> **Формат каждой US:**
> ```
> **US-XXX:** [название]
> **Роль:** [кто]
> **Действие:** [что хочет сделать]
> **Цель:** [зачем]
> **Критерии приёмки:**
> - AC-XXX: **Дано** [контекст], **Когда** [действие], **Тогда** [результат]
> ```
>
> **Ограничения:**
> - Только роль «Клиент»
> - Не добавляй функции вне скоупа (рейтинги, онлайн-оплата)
> - Учти правило 2 часов и статусы: active/cancelled/late_cancel/center_cancelled

**Почему работает:**
- Даёт глоссарий домена (слот, экипировка, типы отмены)
- Задаёт чёткий формат с критериями приёмки Given/When/Then
- Указывает статусы системы (active/cancelled/late_cancel/center_cancelled)
- Ограничивает роль только клиентом
- Запрещает выдуманные функции

**Ответ ИИ (фрагмент):**
> **US-10:** Клиент отменяет запись до старта заезда
>
> **Роль:** Клиент картинг-центра
>
> **Действие:** Хочу отменить свою запись через приложение
>
> **Цель:** Чтобы освободить место для других, если мои планы изменились, и понять последствия отмены
>
> **Критерии приёмки:**
>
> - **AC-001:** **Дано** бронь в статусе `active` и до старта ≥2 часов, **Когда** клиент подтверждает отмену, **Тогда** статус меняется на `cancelled`, места и прокатная экипировка возвращаются в слот, клиент видит снек «Бронь отменена».
>
> - **AC-002:** **Дано** бронь в статусе `active` и до старта <2 часов, **Когда** клиент подтверждает отмену, **Тогда** статус меняется на `late_cancel`, места НЕ освобождаются, штрафов нет, клиент видит снек «Поздняя отмена: место не освобождено. Штраф не взимается».
>
> - **AC-003:** **Дано** бронь в статусе `active` и старт уже наступил, **Когда** клиент открывает детали брони, **Тогда** кнопка «Отменить» недоступна с пояснением «Заезд уже начался — отменить запись нельзя».
>
> - **AC-004:** **Дано** бронь уже отменена (`cancelled`/`late_cancel`), **Когда** клиент открывает детали, **Тогда** кнопка «Отменить» недоступна с пояснением «Запись уже отменена».

## Пример 3: API-спецификация с учётом существующей инфраструктуры

**Промпт:**
> Я описываю REST API для мобильного приложения «Апекс» (картинг-центр).
>
> **Существующая инфраструктура:**
> - Домены API: `auth`, `profile`, `slots`, `bookings`, `marshals`
> - Авторизация: JWT (access_token + refresh_token), OTP по телефону
> - Слоты и маршалы — read-only из существующей админки
> - Бронирования создаются через приложение
>
> **Задача:** Опиши эндпоинт `POST /bookings/{bookingId}/cancel` для отмены брони.
>
> **Формат OpenAPI 3.0:**
> ```yaml
> /bookings/{bookingId}/cancel:
>   post:
>     operationId: cancelBooking
>     summary: ...
>     parameters: ...
>     responses:
>       200: ...
>       409: ...
>       422: ...
> ```
>
> **Бизнес-логика:**
> - Ранняя отмена (≥2ч до старта): статус → `cancelled`, места возвращаются
> - Поздняя отмена (<2ч): статус → `late_cancel`, места НЕ возвращаются
> - Если слот уже стартовал: ошибка 422 `slot_started`
> - Если уже отменена: ошибка 409 `already_cancelled`
>
> **Ограничения:**
> - Метод POST (не DELETE), т.к. отмена — это действие, а не удаление ресурса
> - Время старта в UTC, сравнение на сервере
> - Не добавляй эндпоинты для оплаты (её нет в скоупе)

**Почему работает:**
- Даёт контекст существующей инфраструктуры (домены, авторизация)
- Указывает конкретный формат (OpenAPI 3.0 YAML)
- Описывает бизнес-логику (правило 2 часов, статусы, ошибки)
- Указывает технические детали (POST не DELETE, UTC время)
- Запрещает выдуманные эндпоинты (оплата)

**Ответ ИИ (фрагмент):**
> ```yaml
> /bookings/{bookingId}/cancel:
>   post:
>     operationId: cancelBooking
>     summary: Отмена брони клиентом
>     description: |
>       Отменяет бронь. Тип отмены (ранняя/поздняя) определяется сервером
>       по времени до старта слота (правило 2 часов).
>     security:
>       - bearerAuth: []
>     parameters:
>       - name: bookingId
>         in: path
>         required: true
>         schema:
>           type: string
>           format: uuid
>     responses:
>       '200':
>         description: Бронь успешно отменена
>         content:
>           application/json:
>             schema:
>               $ref: '#/components/schemas/Booking'
>       '409':
>         description: Конфликт — бронь уже отменена
>         content:
>           application/json:
>             schema:
>               type: object
>               properties:
>                 code:
>                   type: string
>                   enum: [already_cancelled]
>                 message:
>                   type: string
>       '422':
>         description: Невозможно отменить — слот уже стартовал
>         content:
>           application/json:
>             schema:
>               type: object
>               properties:
>                 code:
>                   type: string
>                   enum: [slot_started]
>                 message:
>                   type: string
> ```

## Общие правила хороших промптов

1. **Контекст домена.** Дай глоссарий, бизнес-правила, ограничения.
2. **Существующая инфраструктура.** Что уже есть, что нельзя менять.
3. **Формат вывода.** YAML/JSON/таблица, структура, пример.
4. **Ограничения.** Что НЕ делать (не выдумывать, не добавлять).
5. **Уточняющие вопросы.** Разреши ИИ спрашивать, если чего-то не хватает.
6. **Критерии приёмки.** Проси Given/When/Then, а не просто описания.
7. **Роли и скоуп.** Кто пользователь, что входит в MVP.
8. **Технические детали.** HTTP-методы, форматы данных, стандарты.

> **Главное:** Хороший промпт — это не «напиши мне ТЗ», а «вот контекст, вот ограничения, вот формат — помоги структурировать артефакт».
