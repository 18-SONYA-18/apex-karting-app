# Use cases / сценарии

> Этап 2. Сценарии использования. Обязательно прорабатываем **альтернативные потоки**.
>
> **Скоуп: клиентское приложение и API для него.** Слоты, конфигурации трасс и маршалы поступают
> из существующего бэкенда через API (read-only).

---

## UC-1: Запись на заезд (себя и гостей)

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован; в расписании есть хотя бы один слот со свободными местами.
- **Триггер:** Клиент хочет забронировать место на заезде.

> **Бизнес-решение по составу гостей (R-013).** По гостям собираем только **количество мест и выбор экипировки** (своя / прокатная). Имена, контакты, возраст и иные данные гостей в MVP **не собираются**; ответственность за гостей несёт клиент, оформивший бронь.
>
> **Бизнес-решение по идемпотентности (R-022).** Клиент **обязан** передавать `Idempotency-Key` при создании брони. Повтор той же операции (тот же ключ и то же тело запроса) даёт идентичный ответ и не создаёт дубль брони. Повтор с тем же ключом, но другим телом — ошибка.
>
> **Время и источник истины (R-021).** Время хранится и передаётся в UTC, отображается в локальной зоне центра. Сервер — источник истины по времени и статусам; при расхождении клиентского прогноза и серверного ответа применяется серверный статус.

### Основной поток
1. Клиент открывает список слотов.
2. (Опц.) Применяет фильтры: дата/период, конфигурация трассы, наличие мест, маршал.
3. Открывает карточку слота и просматривает параметры (дата/время, конфигурация трассы, маршал, цена, свободно мест, доступность прокатной экипировки).
4. Указывает число мест (себя + до 3 гостей). По гостям указывается только количество (R-013).
5. Для каждого места выбирает вариант экипировки: своя / прокатная.
6. Подтверждает запись. Клиент формирует и передаёт `Idempotency-Key` для этой операции (R-022).
7. Система проверяет лимит мест `max_seats = min(free_seats, track_config.capacity_cap, 4)` (четыре фактора: свободные места, потолок конфигурации трассы, лимит 4 на бронь) и **отдельно** прокатный фонд экипировки `rental_count ≤ free_rental_equipment`, создаёт запись, уменьшает счётчики свободных мест/прокатной экипировки.
8. Система показывает подтверждение с ценой (оплата офлайн) и добавляет запись в «Мои бронирования».

### Альтернативные потоки
- **A1. Вся экипировка — своя.** На шаге 5 прокатный фонд не затрагивается; уменьшается только число мест в заезде.
- **A2. Часть мест — прокат.** На шаге 7 проверяется и уменьшается прокатный фонд экипировки только на число «прокатных» мест.
- **A3. Фильтр не дал результатов.** На шаге 2 список пуст — система предлагает сбросить/изменить фильтры.

### Исключения / ошибки
- **E1. Не хватает мест в заезде.** Запрошено больше, чем свободно мест → запись отклоняется, показывается актуальное число свободных мест.
- **E2. Не хватает прокатной экипировки.** Прокатных мест запрошено больше, чем свободно комплектов экипировки → система предлагает уменьшить число прокатных мест или выбрать «своя экипировка».
- **E3. Гонка запросов / слот заполнился параллельно.** На шаге 7 места уже заняты другим клиентом → запись отклоняется без овербукинга и двойной брони (NFR-8), список обновляется.
- **E4. Сетевой сбой / неопределённый результат при подтверждении (R-020, R-022).** Запись не создаётся частично; при неопределённом результате клиент повторяет операцию с **тем же** `Idempotency-Key`, повтор не приводит к задвоению (NFR-9). Мутации (создание брони) офлайн запрещены — при отсутствии сети показывается понятное сообщение, кнопка подтверждения недоступна.
- **E5. Слот отменён или уже стартовал (R-023).** Слот к моменту подтверждения отменён центром или заезд начался → запись отклоняется с соответствующим кодом (`slot_cancelled` / `slot_started`).

### Матрица ошибок ключевых операций (R-023)

Канон таксономии — коды API. Источник локализованного текста — клиентское приложение (по коду ошибки и полям `details`).

| Код | HTTP | Обязательные `details` | UX-реакция | Retry |
| :-- | :-- | :-- | :-- | :-- |
| `slot_full` | 409 | `available_seats`, `available_rental_equipment` | Показать актуальные свободные места/прокатную экипировку, обновить карточку | Нет (изменить число мест) |
| `double_booking` | 409 | `booking_id` | Сообщить, что бронь на слот уже есть; перейти в «Мои бронирования» | Нет |
| `slot_cancelled` | 410 | `slot_id` | Сообщить об отмене заезда центром, убрать CTA записи | Нет |
| `slot_started` | 422 | `slot_id`, `start_at` | Сообщить, что заезд начался/прошёл, запись недоступна | Нет |
| `invalid_code` | 400/401 | — | Сообщение «неверный код», поле OTP остаётся активным | Да (ввод заново) |
| 401 (unauthorized) | 401 | — | Очистка токена, переход на авторизацию (R-016) | Да (после входа) |
| 403 (forbidden) | 403 | — | Сообщение об отсутствии прав; действие недоступно | Нет |
| 429 (rate limit) | 429 | `retry_after` | Сообщение «слишком много попыток», отложенный повтор | Да (после `retry_after`) |
| 5xx (server error) | 5xx | — | Единый Error/Retry; для брони — повтор по тому же `Idempotency-Key` | Да |

### Постусловия
- При успехе: создана запись клиента; счётчики слота и прокатного фонда экипировки уменьшены; запись видна в «Мои бронирования».
- При ошибке: состояние слота не изменилось.

## UC-2: Отмена записи

- **Актёры:** Клиент
- **Предусловия:** У клиента есть предстоящая запись; заезд ещё не стартовал.
- **Триггер:** Клиент решает отменить участие.

> **Бизнес-решение по объёму отмены (R-014).** Отмена выполняется **только целиком**: отменяется вся бронь со всеми местами и прокатной экипировкой. Частичная отмена (например, одного гостя) **не поддерживается**; при ранней отмене освобождаются все места и прокатная экипировка брони.
>
> **Время и источник истины (R-021).** Время хранится и передаётся в UTC, отображается в локальной зоне центра. Сервер — источник истины по времени и по статусу отмены. Граница политики: **ровно 2 часа до старта = ещё ранняя (бесплатная) отмена**. При расхождении клиентского прогноза (ранняя/поздняя) и серверного ответа применяется **серверный** статус.

### Основной поток
1. Клиент открывает «Мои бронирования» и выбирает предстоящую запись.
2. Нажимает «Отменить» (отмена всей брони целиком, R-014).
3. Сервер определяет время до старта (источник истины — серверное время, R-021).
4. **Ранняя отмена (≥2 ч до старта):** вся бронь помечается «отменена»; **все** забронированные места и прокатная экипировка возвращаются в слот.
5. Система показывает подтверждение отмены.

### Альтернативные потоки
- **A1. Поздняя отмена (<2 ч до старта).** На шаге 4 запись помечается «поздняя отмена»; места НЕ освобождаются; денежных штрафов нет; клиенту показывается пояснение политики 2 часов.
- **A2. Расхождение клиентского прогноза и сервера (R-021).** Клиент показал «ранняя отмена», а сервер вернул «поздняя» (или наоборот) → применяется и отображается серверный статус.

### Исключения / ошибки
- **E1. Заезд уже стартовал/прошёл (`slot_started`).** Кнопка отмены недоступна; запись нельзя отменить.
- **E2. Запись уже отменена (`already_cancelled`).** Повторная отмена не выполняется; показывается текущий статус.
- **E3. Отмена офлайн / неопределённый результат (R-020).** Мутация отмены офлайн запрещена — при отсутствии сети показывается понятное сообщение, кнопка недоступна; при неопределённом результате применяется единый Error/Retry, финальный статус берётся с сервера. Коды ошибок и UX-реакции — по матрице ошибок UC-1 (R-023).

### Постусловия
- Статус всей брони обновлён («отменена» / «поздняя отмена»).
- При ранней отмене все места и прокатная экипировка брони снова доступны другим клиентам.

## UC-3: Просмотр и фильтрация списка слотов

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован.
- **Триггер:** Клиент хочет найти подходящий заезд.

### Основной поток
1. Клиент открывает список слотов. По умолчанию (`only_available=false`) показываются слоты на **ближайшие 7 дней** (дефолт API: `date_from = now`, `date_to = now + 7 дней`; больший период — явным фильтром дат, R-027), отсортированные по времени старта (по возрастанию).
2. Заполненные (`free_seats = 0`) и отменённые слоты отображаются в списке с пометкой «Мест нет» и неактивной (disabled) CTA записи (см. R-012).
3. (Опц.) Задаёт фильтры: дата/период, конфигурация трассы, наличие мест, маршал.
4. Система показывает отфильтрованный список с ключевыми параметрами каждого слота, сохраняя сортировку по времени старта.

### Альтернативные потоки
- **A1. Сброс фильтров.** Клиент очищает фильтры → возвращается дефолтная выдача (слоты на ближайшие 7 дней, дефолт без фильтров).
- **A2. Только свободные слоты.** Клиент включает фильтр наличия мест (`only_available=true`) → заполненные/отменённые слоты скрываются.
- **A3. Семантика комбинирования фильтров (R-026).** Внутри одной группы значения комбинируются по «ИЛИ» (`track_type`, `marshal_id` — массивы), между разными группами — по «И». Границы периода дат включительные; ввод и трактовка дат — в локальной зоне центра. Активные фильтры отображаются клиенту.
- **A4. Пустой справочник маршалов.** Если список маршалов пуст, фильтр по маршалу скрывается.

### Исключения / ошибки
- **E1. Нет слотов под условия.** Показывается пустое состояние с подсказкой изменить/сбросить фильтры.
- **E2. Нет расписания вовсе.** В горизонте выдачи нет ни одного предстоящего слота → показывается пустое состояние «Заездов пока нет».
- **E3. Просмотр офлайн (R-020).** При отсутствии сети показывается ранее загруженный кэш списка с пометкой «данные могли устареть»; единый Error/Retry для повторной загрузки.

### Постусловия
- Клиент видит актуальный (или помеченный как устаревший) список предстоящих слотов; данные расписания не изменяются.

## UC-4: Просмотр истории бронирований

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован; у клиента есть хотя бы одна бронь.
- **Триггер:** Клиент хочет посмотреть свои предстоящие или прошедшие заезды.

### Основной поток
1. Клиент открывает раздел «Мои бронирования».
2. Система показывает список всех броней клиента с пагинацией (`limit`/`offset`), отсортированный по `slot.start_at` (ближайшие выше).
3. Каждая бронь отображается со статусом, параметрами слота (дата/время, конфигурация трассы, маршал), числом мест и вариантом экипировки.
4. Клиент может выбрать конкретную бронь для просмотра деталей.

### Альтернативные потоки
- **A1. Разбиение на предстоящие/прошедшие.** Клиент видит разделение на «Предстоящие» и «История»; разбиение выполняется на клиенте по сравнению `slot.start_at` с текущим временем.
- **A2. Отображение отменённых броней.** В списке отображаются в том числе отменённые брони и поздние отмены (`late_cancel`) с соответствующими статусами.

### Исключения / ошибки
- **E1. Нет броней.** Показывается пустое состояние с подсказкой записаться на заезд.
- **E2. Просмотр офлайн (R-020).** При отсутствии сети показывается ранее загруженный кэш с пометкой «данные могли устареть».

### Постусловия
- Клиент видит свою историю бронирований; данные не изменяются.

## UC-5: Авторизация по SMS OTP

- **Актёры:** Клиент
- **Предусловия:** Клиент не авторизован.
- **Триггер:** Клиент хочет войти в приложение.

### Основной поток
1. Клиент вводит номер телефона.
2. Система отправляет одноразовый код (OTP) по SMS (с учётом rate-limit'ов, NFR-19).
3. Клиент вводит полученный код.
4. Система проверяет код, создаёт/обновляет сессию (JWT: access + refresh токены, NFR-18).
5. Токены сохраняются в защищённом хранилище (Keychain/Keystore).
6. Система переводит клиента на главный экран.

### Альтернативные потоки
- **A1. Повторная отправка кода.** Клиент запрашивает повторную отправку кода (не ранее чем через 30–60 с, NFR-19).
- **A2. Существующий клиент.** Если номер уже зарегистрирован, система просто авторизует клиента без создания нового профиля.

### Исключения / ошибки
- **E1. Неверный код.** Система показывает ошибку «неверный код» (`invalid_code`), поле OTP остаётся активным; число попыток ограничено (≤5, NFR-19).
- **E2. Превышен лимит отправок.** Система показывает сообщение с временем до следующей попытки (`retry_after`, NFR-19).
- **E3. Истёк TTL кода.** Код устарел; требуется запросить новый.
- **E4. Сетевой сбой.** Показывается единый Error/Retry; повторная отправка кода невозможна до восстановления сети.

### Постусловия
- При успехе: клиент авторизован, сессия активна, токены сохранены в защищённом хранилище.
- При ошибке: клиент остаётся неавторизованным.

## UC-6: Управление профилем

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован.
- **Триггер:** Клиент хочет изменить данные профиля или выйти из аккаунта.

### Основной поток
1. Клиент открывает экран профиля.
2. Система показывает текущие данные: имя и номер телефона (замаскирован, NFR-20).
3. Клиент редактирует имя.
4. Система сохраняет изменения.

### Альтернативные потоки
- **A1. Выход из аккаунта (logout).** Клиент нажимает «Выйти»; система очищает токены из хранилища и переводит на экран авторизации (FR-49).
- **A2. Удаление аккаунта.** Клиент инициирует удаление; система:
  - Аннулирует активные брони (освобождает места и прокатную экипировку);
  - Анонимизирует прошедшие брони (сохраняет для истории без ПДн, NFR-20);
  - Удаляет персональные данные (имя, телефон);
  - Переводит на экран авторизации (FR-48).

### Исключения / ошибки
- **E1. Сетевой сбой при сохранении.** Показывается Error/Retry; данные не сохранены.
- **E2. Есть активные брони при удалении.** Система предупреждает, что активные брони будут аннулированы; клиент подтверждает.

### Постусловия
- При успехе: данные профиля обновлены / клиент вышел / аккаунт удалён.
- При ошибке: состояние профиля не изменилось.
